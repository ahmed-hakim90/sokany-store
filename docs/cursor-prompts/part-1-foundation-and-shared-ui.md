You are working inside a Next.js 16.2.3 App Router project with Tailwind CSS v4, TypeScript, React Query, Zustand, and an existing WooCommerce-oriented data layer.

Goal:
Build the visual foundation and shared UI primitives for a full Sokany storefront redesign based on the approved Stitch design. Do not rebuild business logic. Focus only on design tokens, reusable layout shells, and shared UI components that later page work can reuse.

Important constraints:
- Preserve existing data flow and stores.
- Do not break current routes or API behavior.
- Use Arabic RTL by default.
- Support both mobile and desktop.
- Bottom navigation is mobile-only.
- Top header and footer are desktop-first but should still work responsively.
- Use the Sokany visual system:
  - Accent lime: #DAFF00
  - Black: #000000
  - Gray 1: #AFB7B9
  - Gray 2: #D4D4D4
  - Gray 3: #808080
  - White: #FFFFFF
- Prefer Montserrat for UI typography if the project can load it safely with the current Next.js version.
- Avoid hardcoded one-off styling inside pages when it can become a shared primitive.

Before writing code:
- Read the local Next.js docs in node_modules/next/dist/docs relevant to App Router and layout conventions.
- Inspect existing shared components before creating new ones.
- Keep naming consistent with the current repo structure.

- app/globals.css

- components/ui/qty-control.tsx
- components/ui/form-field.tsx

- components/layout/desktop-shell.tsx
- components/layout/page-hero.tsx
- components/layout/section-header.tsx
- components/layout/footer.tsx
- lib/design-tokens.ts (only if useful)

Shared component requirements:

1. Button
- Variants: primary, secondary, ghost, dark
- Sizes: sm, md, lg
- Should support loading and disabled states
- Use shared class composition, not repeated button styles

2. IconButton
- Small square/circle control for menu, search, cart, favorite
- Variants: ghost, subtle, accent
- Sizes: sm, md

3. SearchField
- Reusable for mobile and desktop
- Supports optional leading/trailing icons
- Supports placeholder, controlled value, disabled, and compact mode

4. Badge
- Variants: accent, dark, subtle, success, sale
- Used for offers, labels, and state chips

5. PillFilter
- Horizontal selectable filter chip/button
- Supports active/inactive/disabled

6. Card
- Variants for product, feature, summary, and plain surface
- Shared radius, border, shadow, and background treatment

7. PriceText
- Displays current price and optional old price
- Supports emphasized and compact display
- Respect Arabic formatting direction where appropriate

8. QtyControl
- Minus button, current quantity, plus button
- Supports disabled states
- Shared component, not embedded only inside cart

9. FormField
- Shared labeled input field
- Supports error, helper text, required marker, and RTL labels

10. SelectField
- Shared labeled select/dropdown
- Same surface/states as FormField

Layout component requirements:

1. TopHeader
- Responsive header with logo area, search slot, action icons, and optional nav links
- Mobile and desktop variants through props or composition
- Must not hardcode page-specific content

2. BottomNav
- Mobile-only sticky bottom navigation
- Items: home, categories, cart, about/service, account or more
- Supports active item and highlighted cart action

3. MobileShell
- Shared mobile page shell that composes header, body spacing, bottom nav slot, and floating action slot

4. DesktopShell
- Shared desktop page shell with header, content container, and footer support

5. PageHero
- Reusable hero/banner section for desktop and informational pages
- Supports title, subtitle, image/background, overlay, and optional CTA

6. SectionHeader
- Shared title + subtitle + optional action link pattern

7. Footer
- Rebuild or refactor current footer into the new shared design language without removing useful links

Design implementation notes:
- Create stable CSS variables for color, spacing, radius, and shadows.
- Keep the base interface mostly white/light gray with black typography and lime accents.
- Use lime sparingly for CTA, highlights, active states, and selected filters.
- Keep spacing calm and premium, not crowded.
- Avoid purple, glassmorphism, or generic SaaS styling.

Acceptance criteria:
- Shared primitives exist and are reusable.
- Layout shells exist for mobile and desktop.
- Styling foundation matches Sokany branding.
- Existing app shell still runs.
- No new duplicated button/input/card styling across files.
- `npm run build` succeeds.

When done:
- Summarize which files were added or updated.
- Note any assumptions you made.
